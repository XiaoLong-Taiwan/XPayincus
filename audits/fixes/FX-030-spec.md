# FX-030 规格：礼品卡可无资金来源凭空发行、到账>面值、发行方不可撤销(E4 / BF-8-06/07)

## owner 裁决(BEHAVIOR.md E4)
禁止无资金来源/到账>面值的卡,并允许发行人撤销未兑卡、原额退回。

## 根因
gift-cards.ts(routes:130、150-155、194、439;db:16、227、327):
- 管理员生成礼品卡不扣任何资金账户,"实际到账"可高于面值(单卡≤¥1万、单批≤500 张→单批≈¥500万无来源余额);
- 用户余额生成的卡即时扣款、禁自兑、但无撤销/回收入口,管理员删/停也不退。

## 修法（改 server/src/routes/gift-cards.ts + 必要时 server/src/db/gift-cards.ts;不碰 package.json)
先只读通读礼品卡生成/兑换/删除逻辑与字段(face/面值、actual/到账、status、fundedByUserId/来源、redeemed)。做三件:
1. **禁到账>面值**:生成时若"实际到账额 > 面值"→拒绝(明确 400);到账额恒 ≤ 面值(或直接令到账=面值,取消可配的"到账高于面值")。以"到账 ≤ 面值"为硬约束。
2. **撤销未兑卡、原额退回**:新增/开放"撤销"能力——对**未兑换(unredeemed)**且**由用户余额生成**的卡,允许发行人撤销:卡置为 revoked/void、把当初扣的原额退回发行人余额(走 changeBalance,幂等防重复退)。管理员删除/停用未兑的用户余额卡时**也应退回**(不再"删/停不退")。
3. **无资金来源(管理员凭空发卡)**:先只读确认管理员发卡是否有任何资金账户可扣。
   - 若有(如平台账户)→ 生成时扣该账户,余额不足则拒绝;
   - 若当前架构没有平台资金账户可扣 → **本条不擅自造账户**(属设计变更),改为:①落实第1条(到账≤面值封住"到账虚高");②对管理员批量发卡加合理**单批总额上限**(如单批总面值 ≤ 某可配上限)以限制凭空发行规模;③在交付里明确"管理员发卡的资金来源账户"需 owner 决策(记为设计待办)。
