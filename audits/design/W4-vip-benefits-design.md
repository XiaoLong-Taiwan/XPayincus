## 建议设计(config 驱动,免 schema,待 owner 选权益类型/数值)
### 权益载体
- 新增系统配置 `vip_benefits_config`(JSON):`{ [vipLevel]: { orderDiscountPercent, renewDiscountPercent, extraTrafficPercent, resourcePoolBonusPercent, destroyFeeWaiver, ... } }`;管理端可视化编辑(复用系统配置卡,非锁定表格)。**无需改 schema**(用户 VIP 等级已存)。
### 可选权益类型(供 owner 勾选)
1. **下单/续费折扣 %**(按 VIP 等级递增,如 V1 -2% … V5 -8%)。**关键决策**:与 AFF 码折扣**如何叠加**?建议**取更优惠者**(不叠加)或**相加封顶**——需 owner 定。
2. **额外流量 %**(每周期实例级 extraTrafficQuota 加成,复用 FX-054-rest 加油包机制,一次性/持续?建议持续按等级)。
3. **资源池加成 %**(申领上限提高,复用 FX-072 容量校验)。
4. **销毁手续费减免**(与 D6"一生一次免手续费"协同,VIP 额外减免?)。
5. **风控 QoS 更宽松档**(谨慎,可能被滥用,默认不建议)。
### 强制约束(实现时)
- 折扣类权益必须在**后端下单/续费计算**中生效(不信前端),经守卫钉死;与 AFF/秒杀价(FX-080)/优惠码叠加规则明确单点仲裁。
- 权益随 VIP 等级**实时生效/失效**(降级即失效);不可为负、不可 >100%。
- 加守卫 `test:vip-benefit-route-guards` 断言权益后端生效+叠加规则+范围。
